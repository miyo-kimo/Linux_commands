## SSH Usage

SSH (Secure Shell) is a secure alternative to legacy protocols such as Telnet and Rlogin. It enables encrypted communication between a control node and managed nodes and is widely used in automation tools like Ansible for remote system administration.


<a id="ssh-key-authentication"></a>
## SSH Key Authentication (Public Key vs Private Key)

Ansible typically uses **SSH** to connect from the **Control Node** to **Managed Nodes**. The most secure and recommended way is using **SSH Key Pair** instead of passwords.

### How SSH Key Authentication Works

SSH uses a pair of cryptographic keys generated by **OpenSSH** (the most widely used SSH implementation, originally developed by the OpenBSD project).

You can learn more at:  
- [OpenSSH Official Website](https://www.openssh.com/)  
- [OpenBSD Project](https://www.openbsd.org/)

---

### 1. Private Key
- This is your **secret key**.
- It must stay **only on your Control Node** and never be shared.
- It is automatically created in your home directory under the `~/.ssh/` folder.
- Common filenames: `~/.ssh/id_rsa`, `~/.ssh/id_ed25519`
- It is highly recommended to protect it with a strong passphrase.

### 2. Public Key
- This is the key you can safely share.
- File name ends with `.pub` (e.g. `~/.ssh/id_ed25519.pub`)
- You copy this key to the `~/.ssh/authorized_keys` file on the Managed Nodes.


### Key Concepts

| Item              | Private Key                          | Public Key                              |
|-------------------|--------------------------------------|-----------------------------------------|
| Location          | Only on Control Node                 | Copied to all Managed Nodes             |
| Security          | Must be kept secret                  | Can be shared publicly                  |
| File Name         | `id_rsa`, `id_ed25519`               | `id_rsa.pub`, `id_ed25519.pub`          |
| Purpose           | Proves your identity                 | Allows verification                     |
| Sharing           | Never share                          | Safe to share                           |

---
### Critical Permission Requirements:

- The `~/.ssh` directory should have 700 permissions (drwx------)
- The `authorized_keys` file must have 600 permissions (-rw-------)

If the permissions are too open, SSH will refuse the key and fall back to password authentication (or fail).

<a id="ssh-known-hosts"></a>
## SSH Known Hosts File

When Ansible (or SSH) connects to a remote server for the first time, it verifies the identity of that server to protect against **Man-in-the-Middle (MITM)** attacks.

This information is stored in the **`known_hosts`** file.



### Location

The `known_hosts` file is located in the same folder as your SSH keys:

- For a normal user: `~/.ssh/known_hosts`
- For root user: `/root/.ssh/known_hosts`



### What Does `known_hosts` Do?

- It stores the **public host key** (fingerprint) of every server you have connected to.
- On the first connection, SSH shows you the fingerprint and asks for confirmation.
- After confirmation, the host key is saved in `known_hosts`.
- On future connections, SSH automatically checks the saved key. If it doesn't match, the connection is rejected (protection against spoofing).

<a id="when-known-hosts-is-created"></a>
## When is the `known_hosts` File Created?

The `known_hosts` file is **automatically created or updated on the Control Node** whenever you connect to a remote host **for the first time** using SSH.

It does **not** require `ssh-copy-id`. It is triggered by **any SSH-based connection**.

### Examples that create/update `known_hosts`:

```bash
# These all can create the known_hosts file:
ssh user@192.168.1.100
ssh-copy-id user@192.168.1.100
ansible all -m ping
```

<a id="ssh-keygen-command"></a>
## SSH Key Generation with ssh-keygen

The `ssh-keygen` command is used to generate **SSH key pairs** (private and public keys) for secure, passwordless authentication.


### Basic Usage

```bash
ssh-keygen
```

---
<a id="ssh-copy-id-command"></a>
## Copying Public Key with ssh-copy-id

The `ssh-copy-id` command is a convenient utility that **copies your public SSH key** to a remote server's `authorized_keys` file, enabling passwordless login.


### Basic Usage

```bash
ssh-copy-id <user>@<remote-host-ip>
```

<a id="ssh-debug"></a>
## Debugging SSH Connection with Verbose Mode

After setting up SSH keys, you can verify that Ansible (or SSH) is actually using your **private key** instead of asking for a password by using verbose mode.

### Command

```bash
# Basic verbose (recommended)
ssh -v <user>@<remote-host-ip>

# More detailed (very useful for troubleshooting)
ssh -vv <user>@<remote-host-ip>

# Maximum verbosity
ssh -vvv <user>@<remote-host-ip>
```

### Successful Key Authentication:
``` text
Bashdebug1: Offering public key: /home/user/.ssh/id_rsa RSA SHA256:xxxxxxxxxxxxxxxxxxx 
debug1: Server accepts key: /home/user/.ssh/id_rsa RSA SHA256:xxxxxxxxxxxxxxxxxxx
debug1: Authentication succeeded (publickey)
```
### If it's still asking for password (key not working):
```text
Bashdebug1: No more authentication methods to try.
Permission denied (publickey,password)
```

<a id="authorized-keys-file"></a>
## authorized_keys File

The `authorized_keys` file is one of the most important files in SSH key-based authentication. It contains the **public keys** of all users (or Control Nodes) that are allowed to log in to the server without a password.



### Location

This file is located on the **Managed Nodes** (remote servers) inside the user's home directory:

- For a normal user:  
  `~/.ssh/authorized_keys`

- For the root user:  
  `/root/.ssh/authorized_keys`


### Purpose

- Every time you run `ssh-copy-id`, the public key is appended to this file.
- SSH checks this file to see if the private key presented by the connecting client (your Control Node) matches any of the public keys stored here.
- If a match is found → login is allowed without password.


### Critical Permissions (Very Important!)

SSH is very strict about permissions for security reasons.

```bash
# Correct permissions
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
```


### How to Remove SSH Keys

There are several safe ways to remove public keys from the `authorized_keys` file.



### 1. Clear All Keys (Quick but Dangerous)

```bash
# WARNING: This deletes ALL authorized keys!
> ~/.ssh/authorized_keys
```

<a id="authorized-keys-vs-known-hosts"></a>
## Difference Between `authorized_keys` and `known_hosts`

Although both files are located in the `~/.ssh/` directory, they serve **completely different purposes**.

### 1. authorized_keys

- **Location**: On the **Managed Nodes** (Remote Servers)
- **Purpose**: Controls **who can log in** to this server
- **Contains**: Public keys of allowed clients (Control Nodes)
- **Direction**: Incoming connections (Who is allowed to connect to me?)
- **Managed by**: Server administrator
- **Permission**: `600`

**Example**:  
Your Ansible Control Node’s public key is stored here so it can connect without password.


### 2. known_hosts

- **Location**: On the **Control Node** (Your machine)
- **Purpose**: Helps you verify the **identity of remote servers** (prevents Man-in-the-Middle attacks)
- **Contains**: Host keys / fingerprints of servers you have previously connected to
- **Direction**: Outgoing connections (Which servers have I trusted?)
- **Managed by**: Client (your Control Node)
- **Permission**: `644`

**Example**:  
When you connect to a new server for the first time, its fingerprint is saved in this file.


### Comparison Table

| Feature                    | `authorized_keys`                          | `known_hosts`                              |
|---------------------------|--------------------------------------------|--------------------------------------------|
| Location                  | On Managed Nodes (Servers)                 | On Control Node (Your machine)             |
| Purpose                   | Authentication (Who can log in?)           | Host Verification (Is this server real?)   |
| Contains                  | Client Public Keys                         | Server Host Keys / Fingerprints            |
| Direction                 | Incoming (Server side)                     | Outgoing (Client side)                     |
| Created by                | `ssh-copy-id` or manual                    | Automatically by SSH on first connection   |
| Permission                | `600`                                      | `644`                                      |
| Security Role             | Access Control                             | MITM Protection                            |
